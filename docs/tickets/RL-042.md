# RL-042 — Polish Storytelling: Split Deep Dive Notebook + Consolidate Conclusions & Assumptions

## Status: Done

## Goal
`notebooks/model_deep_dive.ipynb` has grown to 41 cells and ~1.5 MB covering five distinct concerns (controlled comparisons, seed robustness, EVT/XGB audit, rolling backtest, MS-GARCH diagnosis, failure taxonomy, conclusions). It is no longer navigable as a single narrative. Split it into three focused notebooks, each with short inline conclusions, and promote the full prose conclusions and modeling assumptions into dedicated docs.

## Scope

  - §3 Pure Vol Effect (fixed Normal, Constant vs GARCH) at 252-day horizon
  - §3b Pure Vol Effect re-run at 21-day horizon — preserved from source cell 14, with the thin markdown expanded into a full explanation of why 21d is not arbitrary (Basel monthly analogue, RL-038 horizon-crossover evidence, vol-dominant regime for all three assets)

  - §4b Seed Robustness
  - §5 MS-GARCH Model Details

  - §1 Regime stability (HMM flip rate)
  - §2 Per-regime GARCH quality (fallback detection)
  - §3 Breach concentration
  - §4 Per-tier × per-regime breach split (RL-035)
  - §5 When Each Model Fails (failure-mode taxonomy)
  - §6 Overall Conclusions (the full §9 from the source notebook)

- All new notebooks ship with **cleared outputs** — user runs them once, Claude verifies results match the source notebook before the source is deleted.

### RL-040 reframe (not implemented in this ticket, only documented)
Original RL-040 proposed hard-capping Student-t `df` at 3.0 to avoid pathological tail estimates. **Reframed**: do not cap — a fitted `df < 3` is information (the tail is heavier than a finite-variance t can express, GARCH's `σ_lr` is theoretically undefined), not a bug. Surface it as a stability warning instead. Registry gets a one-line note; RL-040 stays Pending with a pointer to this reframe. Implementation is out of scope for RL-042.

## Acceptance criteria
- The three new notebooks exist and each runs end-to-end from a clean kernel in under ~5 minutes total runtime.
- Each notebook is self-contained: opens with its own setup block, requires no prior notebook to have been run, no shared state.
- Numerical results (VaR/ES per tier per asset, backtest breach rates, Kupiec p-values, regime flip rates, per-regime GARCH alphas) match the source `model_deep_dive.ipynb` within seed-level noise.

- `model_deep_dive.ipynb` is deleted only after verification passes.

## Decisions
- **Option 1 (each notebook recomputes), not shared cache.** Self-contained beats fast. Total runtime ~5 minutes is acceptable; pickle-based sharing introduces ordering dependencies and staleness risk that hurt reproducibility more than the time savings are worth.
- **Setup block duplicated, not extracted to `src/analytics/notebook_setup.py`.** Ten lines. Hiding them behind an import makes each notebook harder to read standalone. Matches how `horizon_crossover.ipynb` already handles setup.
- **Inline conclusions short, full prose in `docs/conclusions.md`.** Two-layer structure: notebook readers get the punchline where the evidence is; doc readers get the full story without scrolling through cells. Avoids duplicating full prose in both places while keeping each notebook self-contained as a story.
- **Conclusions in each section, not a terminal §9 cell (except NB3).** NB1 and NB2 integrate conclusions per section because the sections are independent stories. NB3 keeps a terminal conclusions cell because the diagnosis sections build toward a single synthesis — splitting that synthesis across sections would lose the arc.
- **`docs/assumptions.md` is new and overlaps with `docs/decisions/`.** Acknowledged at the top of the file. Reconciliation is explicitly deferred — fixing the overlap now would balloon scope and delay the notebook split, which is the user-facing win.
- **RL-040 reframed, not implemented.** Hard-capping `df` would hide information. Surfacing `df < 3` as a stability warning (per RL-039 transparency philosophy) is the better design but belongs in a separate ticket.
- **Cleared outputs on new notebooks.** User runs them once to populate outputs, Claude verifies numbers against the source before the source is deleted. This prevents a silent copy-paste bug from shipping with stale outputs that look correct.
