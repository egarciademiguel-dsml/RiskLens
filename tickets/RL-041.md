# RL-041 — Model Recommendation Logic in App

## Status: Done

## Goal
Consume the recommendation table produced by RL-038 (`reports/horizon_crossover_results.md`) inside the Streamlit app so the user gets a horizon-aware tier suggestion next to their manual choice.

## Scope
- Embed the recommendation table (asset × horizon → tier) directly in `app/streamlit_app.py`. Hard-code rather than parsing the markdown report — the table is small, fixed, and parsing a doc at runtime is fragile.
- Add a `recommend_tier(ticker, n_days)` helper that:
  - returns the recommended tier label for the three covered tickers (BTC-USD, SPY, NVDA),
  - clamps the requested horizon to the nearest tabulated horizon (1, 5, 10, 21, 63, 126, 252) — anything > 252 maps to the 252d recommendation,
  - returns `None` for tickers not in the table (so the UI can hide the suggestion).
- Show a recommendation panel under the model info box: the recommended tier, whether the user's current pick matches, and (for SPY only) a one-line note that the ranking is non-monotonic across horizons.
- Carry the caveat from the report: "deepest VaR, not best calibrated".
- No analytic module changes. No new files outside the ticket doc.

## Files changed
- `app/streamlit_app.py` — add recommendation table + `recommend_tier` + UI panel.
- `docs/tickets/RL-041.md` (this file).

## Acceptance criteria
- For ticker ∈ {BTC-USD, SPY, NVDA} and any horizon on the slider, the app displays a recommended tier.
- The recommendation matches `reports/horizon_crossover_results.md` exactly for tabulated horizons.
- Horizons above 252d map to the 252d recommendation.
- The panel makes clear when the user's manual choice already matches the recommendation vs differs from it.
- SPY shows a note about non-monotonic behaviour across horizons.
- Tickers outside the covered set produce no recommendation panel (silent, not an error).

## Decisions
- **Hard-coded table, not parsed report.** The recommendation table is 21 cells. Parsing the markdown at runtime would add a parser dependency on the report layout — if we ever reformat the doc, the app would silently break. The table is duplicated, but the duplication is checkable in code review.
- **Clamp to nearest tabulated horizon, not interpolate.** The recommendation is categorical (a tier label), so interpolation is meaningless. For 504/756/1260d we use the 252d row because the long-horizon regime is already dominated by regime mixing — pushing the horizon further does not change which tier is most conservative, only by how much.
- **Tickers outside the set get no recommendation rather than a default.** A blanket default ("use MS-GARCH+EVT for everything beyond 21d") would be wrong on SPY-like assets. Silence is more honest.
- **Surface the SPY non-monotonicity instead of hiding it** — this is the headline finding of RL-038 and a user picking 21d vs 63d vs 126d on SPY should see that the recommendation flips.
- **Caveat is shown inline**, not buried — the recommendation is "deepest VaR at horizon H," not "best calibrated." Without the caveat, the user could mistake conservatism for accuracy.
