# RL-022 — v0.2 wrap: XGBoost vol cutoff, model comparison, code cleanup

## Status: Superseded by RL-023

## Goal
Fix rvol model to be an honest short-term vol forecaster (valid for `horizon` days only), add formal dual model comparison, and clean up architectural issues for v0.2 release.

## What changed

### XGBoost horizon = forecast length
- **`src/analytics/vol_rvol.py`**: `generate_log_returns()` uses XGB-predicted vol for all timesteps (no long-run fallback). Removed `long_run_vol` from `fit_rvol()` result dict.
- **`src/analytics/monte_carlo.py`**: `simulate_paths()` caps `n_days = min(n_days, horizon)` for rvol model. XGB-5d → 5-day simulation, XGB-21d → 21-day simulation. Backtesting `n_days=1` still works (`min(1, horizon) = 1`).

### Model comparison
- **`src/analytics/backtesting.py`**: New `compare_models()` function — runs backtests for all provided models and returns dual ranking: best calibrated (lowest breach_error) and most conservative (lowest breach_rate, deepest mean VaR). `backtest_var()` now tracks `n_skipped` windows (stored in `df.attrs`), surfaced in `backtest_summary()`.

### Code cleanup
- **Global RNG fix**: `monte_carlo.py` replaced `np.random.seed()` (global state pollution) with `np.random.RandomState(seed)` (isolated)
- **Constant consolidation**: `TRADING_DAYS_PER_YEAR` moved to `src/config.py`, imported in `risk_metrics.py`, `monte_carlo.py`, `vol_garch.py` (was duplicated in all three)
- **Dead code removed**: deleted orphan `src/utils.py` (defined `ensure_dirs()`, never imported)
- **Version bump**: `pyproject.toml` updated from 0.1.0 → 0.2.0
- **README rewrite**: accurate test count (253), correct notebook names, full feature list including backtesting/regimes/ML models, updated project structure and tech stack

## Decisions
- Horizon = simulation length: XGB predicts vol for `horizon` days, so that's how far we simulate. No long-run fallback — it was making the XGB model nearly identical to constant vol for long simulations.
- Conservative ranking uses lowest breach_rate + deepest mean predicted VaR — no new metrics needed
