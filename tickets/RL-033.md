# RL-033 — MS-GARCH Fix (Stage 1): Regime Transition Before Return Generation

## Status: Done

## Goal
Eliminate the bug whereby 1-day VaR forecasts from MS-GARCH used only the *current* regime, because the regime transition step in `generate_log_returns` happened *after* the day's return was generated. For `n_days=1`, no path ever sampled tomorrow's regime.

## Scope
- Reorder two steps inside the simulation loop in `src/analytics/ms_garch.py::generate_log_returns`.
- Re-run the BTC backtest and document the improvement.
- Do **not** touch the per-regime GARCH fit (that is RL-034).

## Files changed
- `src/analytics/ms_garch.py` — moved regime transition to the top of the per-day loop.
- `reports/ms_garch_fix_results.md` (new).

## Acceptance criteria
- Day-1 paths sample tomorrow's regime via the HMM transition matrix before generating shocks.
- BTC backtest passes Kupiec.
- No change to the function signature or to any caller.

## Results
| State              | Breach | Kupiec p | Kupiec | Christoffersen p |
|--------------------|--------|---------|--------|------------------|
| Original           | 11.7%  | 0.000   | FAIL   | 0.733            |
| Transition-first   | 7.0%   | 0.123   | PASS   | 0.706            |

## Decisions
- This fix addresses the *temporal misuse* of the model only. The deeper structural issue (alpha=0 in BTC crisis regime) requires refitting GARCH globally — handled in RL-034.
- Statistical pass via Kupiec is necessary but not sufficient: 7.0% is still 40% above target. Stage 2 was scheduled rather than declaring victory here.
