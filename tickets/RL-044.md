# RL-044 — Fix UI Backtest Lookahead Bias with Coarse-Cadence Refit + Session Cache

## Status: Done

## Problem
RL-043 documented that the Streamlit VaR Backtest tab reuses a single upstream fit across every rolling window, introducing **lookahead bias** — the "predicted" VaR at an early test date was computed with parameters that depend on data from after that date. The UI numbers are optimistic relative to a proper walk-forward backtest (the notebook stays strict).

## Solution
Refit the model on a **monthly cadence** (every 21 trading days) instead of every step. Between refits, reuse the most recent fit. This is not lookahead — at every test date `t` the model was fit on data ending at or before `t`. Standard industry practice; matches what risk desks actually do.

Caching moves from (never) to `st.session_state`, keyed by `(ticker, tier, last_date, confidence)`. First visit per combo runs the ~48 refits with a live progress bar; subsequent reruns in the same session are instant.

## Changes

### `src/analytics/backtesting.py`
- Add `refit_every: int = 1` to `backtest_var`. Default preserves strict walk-forward (notebook behavior unchanged).
- Add `progress_callback: callable(i, total) | None = None`. Fires after every test step, including skipped ones.
- Internal: gate the `fit_fn(...)` call on `steps_since_refit >= refit_every`; cache the returned kwargs between refits.

### `tests/test_backtesting.py`
- `test_refit_every_reduces_fit_calls`: with `refit_every=20`, a counting `fit_fn` is called at most `ceil(n_test/20)+1` times (vs. `n_test` with default), and `progress_callback` fires exactly `n_test` times with the final call at `(n_test, n_test)`.

### `app/streamlit_app.py`
- Removed the `fit_fn = lambda rw,s: _bt_kwargs` lookahead shortcut.
- `fit_fn` now dispatches to `constant_fit` / `garch_fit` / `partial(ms_garch_fit, n_regimes=2)` — proper refits.
- Call `backtest_var(..., refit_every=21, progress_callback=_cb)`.
- `_cb` drives `st.progress()` with a per-window counter: *"Backtest MS-GARCH + EVT: window 37/200"*.
- Result cached in `st.session_state[("bt", ticker, tier, last_date, confidence)]` — instant on reruns.
- Added honesty caption on the tab: *"Model is refit every 21 trading days (monthly cadence) and reused between refits. No lookahead. See `notebooks/validation_backtesting.ipynb` for the strict daily-refit reference."*

### `docs/tickets/RL-044.md`, `docs/ticket_registry.md`

## Runtime (measured order-of-magnitude, default 5y data, step=5, refit_every=21 → ~48 refits, ~200 test windows)

| Tier | First run (local) | First run (Cloud) | Cached reruns |
|---|---|---|---|
| Baseline | ~2s | ~2s | instant |
| GARCH+t | ~25–30s | ~45–60s | instant |
| MS-GARCH+EVT | ~1.5–2 min | ~3–4 min | instant |

## Decisions
- **`refit_every=21` (monthly), not 63 (quarterly).** Monthly is closer to daily-refit truth; user picked it over quarterly. Documented here rather than exposed as a UI slider — sliders caused state bugs in the pre-RL-043 version, not bringing them back.
- **`st.session_state` cache, not `@st.cache_data`.** `@st.cache_data` cannot hash a callback function, which we need for the live progress bar. Session-state loses cross-session cache persistence (fine for a single-user demo) and lets us show honest per-window progress on the first run.
- **Default `refit_every=1` in `backtest_var`.** The notebook's strict walk-forward reference must not silently change. UI opts in to the monthly cadence explicitly.
- **Progress fires on skipped iterations too.** A window that fails to fit still counts toward progress; otherwise the bar would stall and the user would think the app hung.
- **Caption on the tab, not a modal/expander.** Short, always visible, disclaims the approximation without being intrusive. Matches the RL-039 transparency philosophy.
