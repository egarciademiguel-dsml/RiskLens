# RL-040 — Surface Student-t df<3 as Stability Warning

## Status: Done

## Goal
When the GARCH+t tier fits a Student-t innovation with `df < 3`, surface it in the app as an explicit stability warning rather than hiding it behind a numerical cap. Below `df=3` Student-t kurtosis is infinite; below `df=2` variance is infinite, at which point GARCH's long-run variance `σ_lr = ω/(1-α-β)` loses its theoretical meaning because the unconditional variance of the innovations it scales is undefined.

## Background / reframe (from RL-042)
The original RL-040 proposal was to hard-cap Student-t `df` at 3.0 to avoid pathological tail estimates. This was reframed in RL-042 as the wrong design: capping **hides information** from the user. A fitted `df = 2.7` (observed on BTC in the current dataset) is a real signal that the empirical tail is heavier than a well-behaved Student-t can express, not a numerical bug to paper over. The correct move — consistent with the RL-039 fit-quality transparency philosophy — is to surface the condition as a warning and steer the user toward the MS-GARCH+EVT tier, which makes no finite-variance assumption.

## Scope
- Use `t_info["df_raw"]` (the unclamped MLE fit) for the check, not the clamped `df` — the 2.1 numerical floor already applied inside `fit_t_distribution` / `vol_garch.generate_log_returns` is a path-stability necessity and must stay, but it should not hide the true fit from the warning logic.
- Two-tier display logic:
  - **`df_raw < 2.0`** → `st.error` ("Danger zone — infinite variance, all tail numbers qualitative only, prefer MS-GARCH+EVT")
  - **`2.0 ≤ df_raw < 3.0`** → `st.warning` ("Stability warning — finite variance but infinite kurtosis, ES/higher quantiles unstable under resampling, prefer MS-GARCH+EVT for tail-sensitive decisions")
  - **`df_raw ≥ 3.0`** → no badge (normal operation)
- Badge appears directly below the existing GARCH+t info box so the context is immediate.
- No change to the simulator. The 2.1 floor inside `fit_t_distribution` stays; this ticket is pure UI transparency.

## Decisions
- **Two tiers (warning vs error), not one.** The difference between `df<3` (finite variance, unstable tail moments) and `df<2` (infinite variance, nothing defined) is large enough to deserve separate UI treatment. A single undifferentiated warning would dilute the `df<2` case.
- **Use `df_raw`, not `df`.** The clamped `df` is what the simulator actually uses for path generation (hard floor at 2.1 to keep variance normalization finite). But the *warning* should reflect what the data actually said, not the floor. Otherwise BTC at true `df≈2.7` would display as the floor value and the warning logic would never fire correctly for asset with true `df<2.1`.
- **Steer toward MS-GARCH+EVT, not away from GARCH+t entirely.** The warning text points the user to the EVT tier because EVT/GPD models the tail non-parametrically and makes no finite-variance assumption. This is the honest recommendation and it aligns with the RL-038 horizon-crossover finding (EVT is the right long-horizon tier anyway).

## Defense value
Without this badge, a reader who opens `docs/conclusions.md` (which already notes BTC df≈2.7) could ask "why is this in the docs but not in the app?" — a credibility hit. With the badge, the app volunteers the caveat itself, which is the stronger posture: the model knows when it is in its danger zone and says so out loud.
