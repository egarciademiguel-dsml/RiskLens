# RL-043 — Retroactive: Streamlit Backtest Tab Stabilization (and the debt it introduced)

## Status: Done (retroactive documentation) — improvements pending

## Context
Three commits on 2026-04-10 landed on `master` without `[RL-XXX]` prefixes:

| Commit  | Message |
|---------|---------|
| `4031bc3` | Remove Tail Risk (EVT & ML) tab — fix app crash blocking VaR Backtest |
| `41e277c` | Auto-run VaR backtest, remove EVT backtest and interactive controls |
| `66823f4` | Speed up backtest: reuse pre-fitted model, reduce MC simulations |

This ticket retroactively documents what they did, why, and the technical debt they introduced — so future work on the backtest UI has the context.

## The problem (as observed on Streamlit Cloud)
The VaR Backtest tab was effectively unusable:
1. **Tail Risk (EVT & ML) tab crashed the app** before the backtest tab could render. EVT's GPD fit raised on short series (<100 obs) and XGBoost CV hyperparameter tuning hung on Streamlit Cloud's constrained runtime.
2. **Interactive controls misbehaved.** The `st.button("Run Backtest")` + two sliders (`bt_window`, `bt_step`) combination caused Streamlit rerun/state issues where clicks would not reliably trigger a run, or stale state would render results from a previous configuration.
3. **Per-window refitting was too slow.** The backtest loop called `garch_fit` / `ms_garch_fit` on every test step — minutes of wall-clock on Cloud, well past user patience and past Streamlit's implicit timeouts.

Net effect: users could not see any backtest graph in the deployed app.

## What was done
### `4031bc3` — surgical removal of the blocker
- Deleted the entire "Tail Risk (EVT & ML)" tab from `app/streamlit_app.py`.
- Rationale at the time: the tab's static VaR comparison added little over the walk-forward backtest, and it was the thing crashing the app. EVT/XGB analysis still lives in `notebooks/validation_backtesting.ipynb`.

### `41e277c` — remove interactivity, auto-run, drop EVT backtest
- Deleted the `st.button("Run Backtest")` gate → tab now auto-runs on every rerender.
- Deleted the `bt_window` and `bt_step` sliders → hardcoded `train_window=252`, `step=5`.
- Removed the secondary "EVT Backtest" section entirely (`backtest_evt_var` import dropped).
- Net: −102 / +51 lines.

### `66823f4` — reuse upstream fit instead of refitting
- Replaced `fit_fn = garch_fit` (or `ms_garch_fit`, or `constant_fit`) with:
  ```python
  fit_fn = lambda rw, s: _bt_kwargs
  ```
  where `_bt_kwargs` carries the **already-fitted** `garch_params` / `ms_garch_params` from upstream in the same Streamlit session.
- Reduced `n_simulations` from 2000 → 500.
- Backtest now completes in seconds.

## Technical debt introduced (⚠ the reason this ticket exists)
**The UI backtest is no longer walk-forward at the fit level.** The model is fit once on the full available series (upstream in the app), and that single fitted parameter set is then applied across every rolling window. This is **lookahead bias**: the "predicted VaR" at an early test date was computed using parameters that depend on data from after that date.

The chart and Kupiec/Christoffersen numbers the UI shows are therefore **optimistic** relative to a true walk-forward backtest. The `validation_backtesting.ipynb` notebook still does a proper per-window refit and is the trustworthy reference.

Secondary issues:
- `n_simulations=500` is borderline for the 5% VaR quantile; adds Monte Carlo noise on top of the lookahead bias.
- Fixed `train_window=252` / `step=5` is not user-tunable.
- No UI indication that the cloud backtest is an approximation.

## Files changed (retroactively attributed)
- `app/streamlit_app.py` — tab removal, control stripping, fit-reuse shortcut, sim count

## Acceptance criteria (retroactive — already met)
- App loads and the VaR Backtest tab renders on Streamlit Cloud.
- Backtest runs to completion in seconds, not minutes.
- Tab shows breach count, Kupiec/Christoffersen p-values, and the returns-vs-VaR chart with breach markers.

## Decisions (as inferred from the commits and outcome)
- **Availability > statistical purity for the deployed demo.** Better to show an approximate backtest than no backtest. Trade-off was made implicitly; this ticket surfaces it.
- **Interactivity removed rather than debugged.** Cheaper than fixing Streamlit state issues for a demo; the cost is users can't explore `train_window` / `step` sensitivity in the UI.
- **EVT backtest dropped from UI entirely.** Lives in notebook; not worth the GPD-fit-fragility in the app.

## Open follow-ups (candidates for the improvement work)
1. **Restore true walk-forward** without bringing the minutes-long runtime back. Options:
   - Refit only at coarser cadence (e.g. every 21 days) with parameter re-use in between — legitimate approximation, standard in practice.
   - Cache refits by `(ticker, end_date)` across Streamlit reruns using `@st.cache_data`.
   - Run the backtest once at app startup and cache the result per ticker; UI just reads.
2. **Surface the approximation honestly.** Small note on the tab explaining the fit-reuse shortcut and pointing to the notebook for the strict walk-forward version.
3. **Reintroduce horizon/window sensitivity** as a read-only toggle (e.g. 126 / 252 / 504) once runtime is under control.
4. **Consider restoring EVT backtest** gated behind a minimum-observations guard (e.g. `if len(returns) >= 500`).
